# AGENTS.md

GitHub-activity-to-Telegram digest bot. Multi-tenant, public repos only, polling-based, digest delivery on per-subscription schedules.

## Stack

Runtime: Bun (latest stable). All scripts run via `bun`.
Language: TypeScript, strict mode on.
Telegram: `grammy` + `@grammyjs/menu` + `@grammyjs/transformer-throttler` + `@grammyjs/auto-retry`.
GitHub: `@octokit/rest` + `@octokit/plugin-retry` + `@octokit/plugin-throttling`.
DB: `drizzle-orm` + `@libsql/client`. Local dev: `file:./dev.db`. Prod: Turso.
Scheduling: `croner`.
In-process queue: `p-queue`.
Logging: `pino` (+ `pino-pretty` dev only).
Container: Docker + Docker Compose, single service.

Do not add: webhooks, Express/Hono/any HTTP server, Redis, BullMQ, conversations plugin, ORMs other than Drizzle.

## Hard architectural constraints

1. Single process. Single Bun runtime. No multi-replica. The bot is the collector, the scheduler, and the deliverer.
2. No public HTTP endpoint. Bot talks to Telegram (long polling) and GitHub (REST). Nothing inbound.
3. Polling only. No GitHub webhooks. No GitHub Apps (deferred to a future rework).
4. Public events only. No PAT/token handling for private events in v1.
5. Cursor-based delivery: a subscription has `lastDeliveredAt`; delivery sends events newer than that and advances the cursor on success. No per-event delivery log.
6. Health and metrics are Telegram admin commands, not HTTP endpoints. Docker `HEALTHCHECK` runs a `bun run healthcheck.ts` script that reads the collector heartbeat from the DB.

## Schema (Drizzle, libSQL)

Use exactly these tables. Field names match.

- `github_accounts`: `id` (PK, GitHub numeric user ID), `login` (current username, unique), `etag`, `lastPolledAt`, `lastEventId`, `consecutiveFailures` (default 0), `pausedUntil` (nullable, set on backoff), `createdAt`.
- `events`: `id` (PK, GitHub event ID string), `accountId` (FK → github_accounts, cascade delete), `type`, `repoName`, `actorLogin`, `payload` (JSON), `createdAt` (GitHub timestamp), `ingestedAt`. Index on `(accountId, createdAt)` and `type`.
- `chats`: `id` (PK, Telegram chat ID, signed 64-bit), `type` ("private"|"group"|"supergroup"|"channel"), `title`, `addedByUserId`, `active` (default true), `banned` (default false), `addedAt`, `deactivatedAt`.
- `subscriptions`: `id` (PK auto), `chatId` (FK → chats, cascade), `accountId` (FK → github_accounts, restrict), `preset` ("firehose"|"releases_only"|"prs_and_releases"|"code_activity"|"new_stuff"|"custom"), `filters` (JSON), `schedulePreset` ("hourly"|"every_6h"|"daily_09"|"daily_18"|"weekly_mon_09"), `timezone` (default "UTC"), `lastDeliveredAt` (nullable cursor), `paused` (default false), `createdAt`, `createdByUserId`. Unique on `(chatId, accountId)`.
- `kv`: `key` (PK), `value`, `updatedAt`. Used for heartbeats and counters.

No `nextDeliveryAt` field — croner owns scheduling state.
No `delivery_log` table — cursor is the source of truth.

## Polling and collection

- Poll interval: every 10 minutes (`*/10 * * * *`), configurable via `POLL_INTERVAL_CRON` env.
- Endpoint: `GET /users/{login}/events`, per `github_accounts` row, with `If-None-Match: <etag>` if etag stored.
- On 304: increment a counter, skip. On 200: parse, filter to events newer than `lastEventId`, insert into `events` table, update `etag` and `lastEventId`.
- On 404 (user deleted/renamed): mark account `consecutiveFailures += 1`. After 5 consecutive failures, set `pausedUntil = now + 24h` and log warn. Re-poll after `pausedUntil` passes.
- On 5xx / network errors: Octokit retry plugin handles automatically. After plugin gives up, increment `consecutiveFailures`.
- Single Croner job runs the collector. Inside, iterate accounts in parallel with concurrency cap of 5 (via `p-queue` instance dedicated to polling, separate from the delivery queue).
- Write a heartbeat to `kv` (`key="collector.last_tick"`, `value=<unix_ms>`) at the end of every successful collector run.

## Delivery

- One Croner job per active subscription. Cron expression derived from `schedulePreset` + `timezone`.
  - `hourly` → `0 * * * *`
  - `every_6h` → `0 */6 * * *`
  - `daily_09` → `0 9 * * *`
  - `daily_18` → `0 18 * * *`
  - `weekly_mon_09` → `0 9 * * 1`
- When a sub is created/updated/paused/resumed/deleted, the deliverer module adjusts (creates/replaces/stops) the corresponding Croner job. Maintain a `Map<subscriptionId, Cron>` in memory.
- When a Croner job fires, push `{ subscriptionId }` onto the delivery p-queue. Queue config: `concurrency: 3`, `timeout: 60_000`, `throwOnTimeout: true`.
- Delivery task:
  1. Load subscription with chat and account joined.
  2. If chat `banned` or `!active`, skip (don't advance cursor, don't log error).
  3. Query events for `accountId` newer than `lastDeliveredAt` (or all if null).
  4. Apply filters (`/filters/apply.ts`).
  5. If 0 events pass: do not send anything. Advance cursor to `now`. Log at debug.
  6. If >0: render with `/formatting/render.ts`. Send to chat. On success, advance cursor to the newest event's `createdAt`.
  7. On Telegram send failure: log error, do NOT advance cursor. The next scheduled run will retry.
- Write a heartbeat (`key="deliverer.last_tick"`) after each delivery task settles (success or failure).

## Filters

Filter JSON shape (validated with zod on save):

```ts
{
  events: ("push"|"pull_request"|"issues"|"release"|"repository"|"fork"|"star"|"create")[],
  repos: { include: string[], exclude: string[] },     // globs, default { include: ["*"], exclude: [] }
  ignoreBotAuthors: boolean,                            // default true
  minCommitsPerPush: number,                            // default 1
  branches: { include: string[], exclude: string[] }    // globs, default { include: ["*"], exclude: [] }
}
```

Presets (`/filters/presets.ts`):

- `firehose`: all events, all repos, ignoreBotAuthors: false
- `releases_only`: events: ["release"]
- `prs_and_releases`: events: ["pull_request", "release", "repository"]
- `code_activity`: events: ["push", "pull_request"], branches.include: ["main", "master"], ignoreBotAuthors: true
- `new_stuff`: events: ["repository", "release", "fork", "star", "create"]

When a user toggles a filter in the menu away from preset defaults, set `subscriptions.preset = "custom"`. "Reset to preset" sets it back and restores the preset's JSON.

## Telegram bot configuration

- Use HTML parse mode globally. Escape user-controlled strings (repo names, commit messages, usernames) with a helper that escapes `<`, `>`, `&`.
- Throttler: install `@grammyjs/transformer-throttler` with default config on `bot.api.config.use`.
- Auto-retry: install `@grammyjs/auto-retry` with `maxRetryAttempts: 3, retryOnInternalServerErrors: true`.
- Menu: use `@grammyjs/menu` for every interactive surface. No callback_query handlers outside the menu plugin.
- On `my_chat_member` updates: if status is "left" or "kicked", mark chat inactive. If status is "member" or "administrator" and chat is new, create the row.

### Button color conventions

The bot uses native button styles (Bot API 9.4+, Feb 2026) via grammY's `.danger()`, `.success()`, `.primary()` methods on `Menu`/`MenuRange`. Used sparingly — meaningful, not decorative.

Rules:

- `.success()` (green): confirming/saving/positive actions. Examples: Save filters, Resume subscription, Enable filter.
- `.danger()` (red): destructive/irreversible actions. Examples: Delete subscription, Unsubscribe.
- `.primary()` (blue): primary CTA on a screen. Examples: Add new subscription, Preview now.
- No style (default): toggles, navigation (Back), reversible state changes (Pause).
- Maximum one `.success()`, one `.danger()`, one `.primary()` per menu screen. If a screen would have more, the screen is doing too much — split it.
- Filter toggles use `☑️`/`☐` in the label; do not also add `.success()` to toggled-on buttons.

### Commands

Only these commands exist. Do not add more without explicit instruction.

Public:

- `/start` — onboarding text. In groups, briefly explains the bot; in DMs, fuller intro.
- `/help` — short list of available commands and one-line description of each.
- `/ping` — public: "alive, last collector tick Ns ago". Admin (in ADMIN_IDS): adds rate limit remaining, active subs, active chats, events ingested in last 24h, errors in last 24h.

Chat-scoped:

- `/subscribe` — no args: opens root menu (list existing subs in chat + "Add new"). With `<github_username>`: opens subscription menu for that user, creating a draft if none exists.

Admin (ADMIN_IDS only):

- `/admin` — opens admin menu (Chats list, Accounts list, Broadcast, Diagnostics, Force-poll, Force-deliver).

All chat-scoped mutations (creating, editing, pausing, deleting subscriptions) require the calling user to be an admin of the chat (or in `ADMIN_IDS`). Verify via `getChatAdministrators`. DMs implicitly grant admin rights to the DM owner.

### Schedule menu

Radio-style list of preset options. Selecting one stages the change; Save commits.

### Timezone menu

Top: 6 common IANA zones (UTC, America/Santiago, America/New_York, Europe/London, Europe/Madrid, Asia/Tokyo). Below: "Other..." which opens an ad-hoc text-input flow with the same 60s-TTL pattern as username entry. Validate input against IANA list (use Intl.supportedValuesOf("timeZone")).

### Admin menu (entry: `/admin` from ADMIN_IDS)

Buttons for: Chats (paginated list), Accounts (paginated list), Broadcast (text-input + confirm), Diagnostics (same content as admin `/ping`), Force-poll (account picker), Force-deliver (sub picker). Destructive admin actions use `.danger()`. The broadcast confirm step uses `.danger()` for "Send" because broadcasting to all chats is irreversible.

## Logging

- Root logger: pino, level from `LOG_LEVEL` env (default "info"). Pretty-print in dev (`NODE_ENV !== "production"`).
- Structured logs only: `logger.info({ chat_id, sub_id, event_count }, "delivery complete")`. Never string interpolation of fields.
- Child loggers: every delivery task creates a child with `{ subscription_id, chat_id, account_login }`. Pass the child through; do not import root logger in deeply nested code.
- Levels:
  - `trace`: per-event filter decisions
  - `debug`: per-account poll outcomes (including 304s), per-delivery filter-result counts
  - `info`: subscription created/updated/deleted, deliveries sent (with event count and duration), collector tick summary
  - `warn`: account auto-paused after consecutive failures, rate-limit approaches, Telegram 429 from a chat
  - `error`: delivery send failure, DB write failure, unhandled exceptions in scheduled jobs
- Never log full GitHub event payloads at info level. Payloads at debug only. Always log event IDs and types.
- Never log bot tokens, Turso auth tokens, or full request URLs containing tokens.

## Metrics (in-memory counters)

Increment via a single `/lib/metrics.ts` module exporting typed counter functions. Read by admin `/ping` and `/admin` Diagnostics.

Counters:

- `github_api_requests_total{status: "200"|"304"|"4xx"|"5xx"|"error"}`
- `events_collected_total{type}`
- `deliveries_sent_total{status: "ok"|"empty"|"error"}`
- `delivery_duration_ms` (rolling histogram, last 100)
- `telegram_api_errors_total{code}`
- `subscriptions_active` (gauge, recomputed on read)
- `chats_active` (gauge, recomputed on read)

## Env vars

Validated with zod at startup. Crash on missing required.

Required:

- `BOT_TOKEN` — Telegram bot token
- `ADMIN_IDS` — comma-separated Telegram user IDs
- `DATABASE_URL` — `file:./dev.db` or `libsql://...`
- `DATABASE_AUTH_TOKEN` — required if `DATABASE_URL` starts with `libsql://`

Optional with defaults:

- `LOG_LEVEL` (default "info")
- `NODE_ENV` (default "development")
- `POLL_INTERVAL_CRON` (default `*/10 * * * *`)
- `MAX_SUBS_PER_CHAT` (default 20)
- `GITHUB_TOKEN` — if set, used to authenticate GitHub requests for the higher rate limit. Still public events only.

`/.env.example` lists all of these with safe placeholder values.

## Shutdown

Single `setupShutdown()` function called from `/src/index.ts`. Behavior:

1. Trap `SIGINT` and `SIGTERM` once each.
2. Set a `shuttingDown = true` flag (exported from `/lifecycle/shutdown.ts`) that the collector and delivery loops check at the top of each iteration.
3. Stop all Croner jobs (`.stop()` each one in the in-memory Map).
4. Drain the p-queue (`await queue.onIdle()`).
5. Call `await bot.stop()` (waits for in-flight grammY updates).
6. Close the libSQL client.
7. `process.exit(0)`.

`stop_grace_period: 30s` in compose. If shutdown exceeds 25s, log warn and force-exit with code 1.

## Docker

Single-service compose. The volume mounts `./data` for the local SQLite file in dev (in prod the bot connects to Turso and the volume is unused but kept for portability).

`Dockerfile`: multi-stage, Bun base image (`oven/bun:1-alpine` or current latest tag), production deps only in final stage, non-root user. `HEALTHCHECK` runs `bun run healthcheck.ts` every 60s with 3 retries.

`healthcheck.ts` exits 0 if `kv["collector.last_tick"]` is within `2 * POLL_INTERVAL_MINUTES` of now, else exits 1.

## Code style

- TypeScript strict mode. No `any` in production code. `unknown` + narrowing is fine.
- No default exports. Named exports only.
- All DB access goes through `/db/queries.ts`. Other modules import named query functions, not the `db` instance directly.
- All logging through `/lib/logger.ts`. No `console.*` in production code.
- All env access through `/lib/env.ts`. No `process.env.*` outside that module.
- No em dashes in user-facing strings or comments.
- No AI-flavored words: don't write "leverage", "robust", "seamless", "delve", "ship", "comprehensive solution". Plain conversational English.
- Comments explain why, not what. No JSDoc except on exported public APIs.
- Imports sorted: node built-ins, then external packages, then `~/` aliases.
- File names: kebab-case for files, PascalCase for class names, camelCase for everything else.

## What not to do

- Do not add a web framework or HTTP endpoint.
- Do not add Redis, BullMQ, or any external queue.
- Do not add `@grammyjs/conversations`. Use the 60s-TTL Map pattern for the two text-input cases (username, custom timezone).
- Do not add a per-event `delivery_log` table. The cursor is the source of truth.
- Do not write inline SQL outside `/db/queries.ts`.
- Do not add support for private GitHub events, GitHub Apps, or OAuth in v1.
- Do not introduce multi-replica logic, leader election, or distributed locks.
- Do not emit any external HTTP request that is not to `api.telegram.org` or `api.github.com`.

## Testing

- Use Bun's built-in test runner (`bun test`).
- Unit-test filter application (`/filters/apply.ts`) with hand-written event fixtures for each event type.
- Unit-test the formatter (`/formatting/render.ts`) with fixtures, snapshot the HTML output.
- Integration test the polling loop against a recorded fixture (no live GitHub calls in tests).
- Do not write end-to-end tests against live Telegram.

## Definition of done for v1

- `/start`, `/help`, `/ping`, `/subscribe`, `/admin` all work as specified.
- A user can `/subscribe @someuser` in a group, pick a preset, save, and receive a digest at the scheduled time.
- A second chat subscribing to the same `@someuser` does not cause a second poll.
- Bot survives Telegram 429, GitHub 5xx, and a 60s network outage without crashing or losing the cursor.
- `docker compose up` brings the bot online with the env file populated.
- Admin can `/ping` and see live counters; admin can `/admin` to broadcast a message to all chats.
